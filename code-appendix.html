Code Appendix: Pedagogical Implementations

These are original, minimal, illustrative Python/NumPy-style implementations of the core mechanisms discussed in Chapters 9, 13–16, and 19 — written for this site to make each mechanism concrete, not copied from the source PDF and not intended for production use. Every snippet intentionally omits batching edge cases, numerical-stability hardening beyond the minimum needed to illustrate the idea, autograd/framework-specific plumbing, and performance optimization. Each card states the one architectural idea it demonstrates before the code.

Read this before the code

None of the code on this page is a production kernel. A real FlashAttention or Mamba implementation uses hand-tuned, hardware-specific kernels far more complex than the versions below; these snippets exist purely to make the underlying algorithmic idea legible in plain Python, matching the source's own framing of its code appendix as illustrative rather than production code.

E.1 Scaled Dot-Product & Multi-Head Attention

Demonstrates the mechanism from Chapter 9 (The Transformer).

Architectural idea: attention lets every position compute a weighted average over every other position's value vector, with weights determined by a content-based query/key match rather than fixed position — the core operation that replaced recurrence.

E.2 Causal Mask & RoPE

Demonstrates mechanisms from Chapters 9 and 13.

Architectural idea: a causal mask enforces that position t can only attend to positions ≤ t, which is what makes a Transformer a valid autoregressive language model; RoPE rotates query/key vectors so their dot product encodes relative, not absolute, position.

E.3 RMSNorm & SwiGLU

Demonstrates mechanisms from Chapter 13 (Optimization & Training Stability).

Architectural idea: RMSNorm normalizes only by magnitude (no mean-centering, no bias); SwiGLU multiplies a gate branch elementwise into a value branch before the output projection, adding a multiplicative interaction a plain ReLU MLP lacks.

E.4 Tiled Online-Softmax Attention

Demonstrates the FlashAttention idea from Chapter 15 (Long-Context Architectures).

Architectural idea: by maintaining a running maximum and running denominator, softmax over a whole row can be computed exactly from small tiles processed one at a time, without ever holding the full row of scores in memory at once — this is the algorithmic core of FlashAttention, shown here as an illustrative loop rather than a real SRAM/HBM-aware kernel.

E.5 Top-k MoE Router & Load Balancing

Demonstrates mechanisms from Chapter 14 (Mixture-of-Experts).

Architectural idea: a router scores every token against every expert, keeps only the top-k per token, and combines the selected experts' outputs weighted by (renormalized) router probability; a load-balancing loss discourages the router from collapsing onto a small subset of experts.

E.6 Mamba-Style Selective Scan

Demonstrates mechanisms from Chapter 16 (State-Space Models).

Architectural idea: a state-space model updates a fixed-size hidden state each step via a linear recurrence; making the recurrence parameters input-dependent ("selective") lets the model decide, per token, how much to write into or forget from that state.

E.7 Sliding-Window KV Cache

Demonstrates mechanisms from Chapter 15 (Long-Context Architectures).

Architectural idea: rather than let the KV cache grow forever, a sliding-window cache keeps only the most recent W tokens' keys/values, bounding serving memory at the cost of losing exact access to anything older than the window.

E.8 Cosine Warmup Schedule & Chinchilla Solver

Demonstrates mechanisms from Chapter 13 (Optimization) and Chapter 12 (Scaling Laws).

Architectural idea: a short linear warmup avoids destabilizing a randomly initialized network with a large learning rate, followed by a cosine decay to a small final value; the Chinchilla solver recovers a compute-optimal parameter/token split from a fixed compute budget and a fitted tokens-per-parameter ratio (see Appendix D.3 for the same arithmetic worked by hand).

E.9 Minimal Pre-Norm Transformer Block

Demonstrates the assembled block from Chapters 9 and 13.

Architectural idea: a full "modern" Transformer block is just the pieces above (attention, RMSNorm, SwiGLU) composed around two residual connections, each with pre-norm placement — nothing in this assembly step introduces new mathematics beyond what E.1–E.3 already defined.

Key insight

Nothing in tiny_lm_forward is specific to any one chapter — it is literally Chapter 9's Transformer block (attention + feed-forward + two residual connections) with Chapter 13's specific ingredient choices (RMSNorm instead of LayerNorm, SwiGLU instead of ReLU) substituted in. Seeing the assembly this way is the clearest illustration of the book's own claim that "modern" Transformers are a base architecture plus a set of targeted, swappable efficiency choices.
